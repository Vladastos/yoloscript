# 0001 — v0.1 Feature Set

- **Date:** 2026-04-04
- **Status:** Accepted

## Context

We need a concrete, implementable target for the first working version of Yolang — an interpreter that can run real programs. The goal of v0.1 is not completeness; it is to establish a **stable, self-consistent core** that will not need to be revisited or redesigned as the language grows. Every feature included must be something we are confident we would not need to change in a breaking way later.

Features that are powerful but whose design is still open (trait objects, string interpolation syntax, derive macros, module system) are explicitly deferred. They should not block a working interpreter.

The guiding question for inclusion was: *"If we implement this now, will we regret the design decision later?"* If the answer was uncertain, the feature was deferred.

---

## Included features

### Lexical

- Single-line (`//`) and block (`/* */`) comments. Block comments do not nest.
- Identifiers: `[a-zA-Z_][a-zA-Z0-9_]*`
- Integer literals (decimal, with `_` separators): `42`, `1_000_000`
- Float literals: `3.14`, `2.0`
- String literals: double-quoted UTF-8, with `\n \t \\ \" \r` escapes
- Boolean literals: `true`, `false`
- The absence literal: `nope`

### Types

- Primitive types: `Int` (64-bit signed), `Float` (64-bit), `Bool`, `String`, `()`
- Local type inference for `let` and `mut` bindings
- Explicit type annotations required on function parameters, return types, struct fields, and enum fields
- Generic types and functions with `<T>` syntax
- `where` clauses and inline trait bounds (`<T: Trait>`)
- `Perhaps<T>` as the built-in optional type (no null anywhere)
- `Result<T, E>` as the built-in fallible-operation type
- `T[]` / `Array<T>` as the built-in fixed-capability sequence type (see Arrays section below)
- Tuples: `(Int, String)` syntax, positional access via `.0`, `.1`, ..., and destructuring in `match`
- Type casting via `as` for numeric primitives: `Int`, `Float`. The `as` expression desugars to a call to the `From` trait; casting is infallible and returns the target type directly. Example: `let f: Float = x as Float;`

### Variables

- `let x = expr;` — immutable binding, must be initialized
- `mut x = expr;` — mutable binding, may be reassigned
- Lexical scoping; inner scopes shadow outer names
- Variable shadowing is allowed

### Functions

- Named functions: `fun name(params) -> ReturnType { body }`
- No return type annotation means the function returns `()`
- `return expr;` and bare `return;`
- Generic functions with type parameters
- Functions are first-class values (can be assigned and passed)
- Closures: `fun(params) -> ReturnType { body }` in expression position
- Closures capture variables from their enclosing scope
- Closure type in signatures: `fun(Int) -> Int` (same syntax as function types)
- Associated functions (no `self`) inside `impl` blocks — the canonical constructor pattern `Type::new(...)`

### Structs

- `struct Name { field: Type, ... }`
- Struct literal construction: `Name { field: expr, ... }`
- Field access: `value.field`
- Generic structs: `struct Pair<A, B> { ... }`
- `impl Type { fun method(self, ...) { ... } }` for methods
- `mut self` for methods that mutate the receiver

### Enums

- `enum Name { Variant, Variant { field: Type, ... }, ... }`
- Unit variants and struct-like data-carrying variants
- Enum construction: `Name::Variant` or `Name::Variant { field: expr, ... }`
- `impl` blocks on enums (same syntax as structs)
- Generic enums

### Pattern matching

- `match expr { pattern => expr, ... }`
- Patterns: wildcard `_`, literal (`Int`, `Float`, `String`, `true`, `false`, `nope`), binding (bare identifier), enum variant with optional field destructuring, tuple destructuring
- Guards: `pattern if condition => expr`
- `match` is an expression (its arms must produce the same type)
- Exhaustiveness is checked — every possible case must be covered

### Control flow

- `if (cond) { } else if (cond) { } else { }` — both statement and expression form
- `while (cond) { }`
- C-style `for (mut i = 0; i < 10; i = i + 1) { }`
- Iterator `for (let item in collection) { }`
- `loop { }` — infinite loop
- `break` — exit the innermost loop; `break expr` to produce a value from a `loop` expression
- `continue` — skip to the next iteration of the innermost loop
- Compound assignment operators: `+=`, `-=`, `*=`, `/=`, `%=`

### Operators

| Category        | Operators                                    |
| --------------- | -------------------------------------------- |
| Arithmetic      | `+`  `-`  `*`  `/`  `%`                      |
| Compound assign | `+=`  `-=`  `*=`  `/=`  `%=`                 |
| Comparison      | `==`  `!=`  `<`  `<=`  `>`  `>=`             |
| Logical         | `&&`  `\|\|`  `!`  (`and` / `or` as aliases) |
| Assignment      | `=`                                          |
| Error prop      | `?`                                          |
| Type cast       | `as`                                         |
| Path            | `::`                                         |
| Range           | `..`  `..=` (for use in `for-in` only)       |

### Traits

- `trait Name { fun method(self, ...) -> ReturnType; }` with optional default implementations
- `impl Trait for Type { ... }`
- Trait bounds in generics
- `Self` type inside trait definitions
- No trait objects (`dyn Trait`) in v0.1 — only static dispatch

### Error handling

- `?` operator on `Result<T, E>` — desugars to: if `Err`, return the error; if `Ok`, unwrap the value. The function's return type must be `Result<_, E>` and the error types must match exactly (no implicit coercion in v0.1)
- `.yolo()` — unwrap a `Perhaps<T>` or `Result<T, E>`, panic at runtime if absent/error

### Arrays

- `Array<T>` (also written `T[]`) is the built-in ordered sequence type. It is a low-level, resizable sequence managed by the runtime.
- Literal syntax: `[1, 2, 3]` — the type is inferred as `Int[]`
- Index access: `arr[i]` where `i: Int`; out-of-bounds causes a panic
- Usable in `for-in` loops
- `List<T>` is **not** a built-in — it is a standard library type built on top of `Array<T>`, available in v0.1 as a global built-in (pre-imported) but not a language primitive

### Standard library (built-ins only, no module system needed)

These are available globally without any `use` declaration in v0.1:

| Name              | Signature                          | Description                              |
| ----------------- | ---------------------------------- | ---------------------------------------- |
| `print`           | `(value: String)`                  | Print a string to stdout, no newline     |
| `println`         | `(value: String)`                  | Print a string to stdout with newline    |
| `int_to_string`   | `(n: Int) -> String`               | Convert `Int` to its decimal string form |
| `float_to_string` | `(n: Float) -> String`             | Convert `Float` to its string form       |
| `bool_to_string`  | `(b: Bool) -> String`              | `"true"` or `"false"`                    |
| `string_len`      | `(s: String) -> Int`               | Number of characters                     |
| `string_concat`   | `(a: String, b: String) -> String` | Concatenate two strings (also via `+`)   |
| `array_push`      | `(arr: T[], value: T)`             | Append a value (mutates the array)       |
| `array_len`       | `(arr: T[]) -> Int`                | Number of elements                       |
| `clock`           | `() -> Int`                        | Unix timestamp in milliseconds           |

### Panics

A panic is a hard, unrecoverable runtime error. It prints a message and exits the process with a non-zero status. There is no way to catch a panic in v0.1. Panics are triggered by:

- `.yolo()` on `nope` or an `Err` variant
- Out-of-bounds array access
- Integer division by zero
- Explicit runtime errors (reserved for the runtime)

## Explicitly deferred (not in v0.1)

The following items from the roadmap are consciously excluded. They are not missing by accident — they are deferred because their design is not yet settled enough to implement without risking revision.

| Feature                     | Reason for deferral                                                           |
| --------------------------- | ----------------------------------------------------------------------------- |
| Module system / `use`       | Design is open; not needed when everything is in one file                     |
| Visibility (`pub`)          | Depends on module system                                                      |
| `UInt` type                 | Adds complexity to casting and arithmetic; `Int` is sufficient for v0.1       |
| String interpolation        | Syntax not decided; `int_to_string` + `+` is sufficient for v0.1              |
| Trait objects (`dyn Trait`) | Fat-pointer semantics not designed; static dispatch covers all v0.1 use cases |
| Derived / auto-impl traits  | Syntax not designed (`#[derive]` is very Rust; want something different)      |
| Operator overloading traits | Can be added later without breaking existing code                             |
| `?` error type coercion     | `From`-style coercion deferred; same error type is fine for v0.1              |
| `pub use` re-exports        | Depends on module system                                                      |
| Integer overflow behavior   | Defined as wrapping for now, to be revisited                                  |

## Consequences

- A v0.1 interpreter can be written against this feature set without needing to revisit core decisions.
- All programs must be single-file in v0.1 (no module system).
- String formatting is verbose but workable via `+` and conversion functions.
- The grammar in the spec needs to be updated to reflect tuples, `Array` / `T[]` literals and type syntax, `loop`, `break`, `continue`, compound assignment, and `as` casting (which produces a `Result`).
- The roadmap items covered here should be marked as resolved for v0.1.
- Items in the "deferred" table remain open in the roadmap for v0.2+.
